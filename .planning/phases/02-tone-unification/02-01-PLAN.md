**Test suite structure:**
1. **No PM language test** — Scans all prompts for banned terms: "dev team", "your team", "stakeholder", "leadership", "product roadmap", "manager", "standup"
2. **Inline definition test** — Validates that common tech terms (when used) include inline parenthetical definitions or are avoided entirely
3. **Analogy presence test** — Confirms at least one everyday analogy exists in explain/bugs/refactor prompts (using keyword detection: "like a", "think of", "imagine")
4. **Personality archetype test** — Validates each mode establishes distinct identity:
   - explain: "patient teacher" or "colleague"
   - bugs: "protective" or "safety inspector"
   - refactor: "coach" or "mentor"
   - translate modes: "translator" or "bridge"
5. **Guardrail present test** — Confirms MODE_GUARDRAIL is appended to all non-review modes

Use Node.js `assert` module for assertions. Import SYSTEM_PROMPTS from `../lib/prompts.js` using CommonJS require.

**Test file template:**
```javascript
const { describe, it } = require('node:test');
const assert = require('node:assert');
const { SYSTEM_PROMPTS } = require('../lib/prompts');

describe('Tone Validation', () => {
  const PM_TERMS = ['dev team', 'your team', 'stakeholder', 'leadership', 'product roadmap', 'manager'];

  it('should not contain PM-specific language', () => {
    for (const [mode, prompt] of Object.entries(SYSTEM_PROMPTS)) {
      if (mode === 'review' || mode === 'review-fallback') continue; // Phase 1 prompts
      const lower = prompt.toLowerCase();
      for (const term of PM_TERMS) {
        assert.ok(!lower.includes(term), `${mode} prompt contains PM term: "${term}"`);
      }
    }
  });

  // Additional tests for inline definitions, analogies, personality, guardrail
});
```

File should be runnable with: `node --test tests/tone-validation.test.js`

**IMPORTANT:** Tests will FAIL initially (prompts still PM-focused). This is Wave 0 — scaffold only. Task 1 will fix the prompts.

**Modes to rewrite:**

1. **chat** — Remove PM context, change to vibe-coder framing

2. **explain** — Patient teacher for non-coders (TONE-02)
   - Identity: "friendly, patient teacher helping someone who has never written code"
   - Analogies: Use everyday life examples (kitchen, library, building)
   - Structure: Keep ## What This Does, ## How It Works, ## Why It Matters
   - Change "Why It Matters for Your Product" → "Why It Matters for Your Project"

3. **bugs** — Protective friend with plain-English severity (TONE-03)
   - Identity: "thoughtful, supportive safety inspector" (keep this — already vibe-coder friendly)
   - Remove: Any PM-specific references
   - Keep: "What could users actually experience" framing (perfect for vibe coders)
   - Severity framing: Already plain-English, no changes needed

4. **refactor** — Helpful coach with AI prompts (TONE-04)
   - Identity: "supportive coding mentor" (keep)
   - ADD NEW SECTION after showing improved code:
   ```
   ## Here's What to Tell Your AI
   Copy-pasteable prompts you can use in Cursor, ChatGPT, or your coding tool:
   - "Refactor this code to [specific improvement]"
   - "Update the variable names to be more descriptive"
   ```
   - Keep: Show improved code first, then explain changes

5. **translate-tech** — Code to plain English (TONE-05)
   - Remove: "stakeholder", "business side", "engineering team"
   - Change "Your Talking Points" → "How to Explain It" with simpler framing: "2-3 simple talking points if someone asks what this does"
   - Keep structure but shift from PM→Business to Anyone→Plain English

6. **translate-biz** — Idea to code spec (TONE-05)
   - Remove: "tech lead", "engineering team", "PM"
   - Change "What Needs to Be Built" → "What to Tell Your AI Coding Tool"
   - Reframe as: helping user translate their idea into instructions for their AI tool
   - Change "Things I'd Want to Clarify" → "Things to Think About"
