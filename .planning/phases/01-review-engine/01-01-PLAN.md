  <name>Task 1: Create report card Zod schema with JSON Schema export</name>
  <files>lib/review-schema.js</files>
  <action>
Create `lib/review-schema.js` using CommonJS (`require`/`module.exports`) to match existing backend conventions.

Define the report card schema using Zod v4 (already installed at ^4.3.6):

1. **GradeEnum**: `z.enum(['A', 'B', 'C', 'D', 'F'])` — letter grades only, no +/- variants.

2. **FindingSchema**: Object with fields:
   - `title` (string) — short name for the issue
   - `severity` (enum: 'critical', 'high', 'medium', 'low')
   - `explanation` (string) — plain-English explanation, no jargon. Per CONTEXT.md: short (2-3 sentences) for minor, longer (paragraph + analogy + action) for critical
   - `analogy` (string, optional) — everyday analogy. Per CONTEXT.md: for critical/high severity ONLY

3. **CategorySchema**: Object with fields:
   - `grade` (GradeEnum)
   - `summary` (string) — one-sentence category summary
   - `findings` (array of FindingSchema)

4. **ReportCardSchema**: Object with fields:
   - `overallGrade` (GradeEnum) — overall code quality grade
   - `topPriority` — object with `category` (string), `title` (string), `explanation` (string: "Why this is the #1 thing to fix, plain English")
   - `categories` — object with keys: `bugs`, `security`, `readability`, `completeness`, each a CategorySchema
   - `cleanBillOfHealth` (boolean) — true when all categories are A grade (triggers celebration in UI per CONTEXT.md)

5. Generate JSON Schema using `z.toJSONSchema(ReportCardSchema)` and export it.

Export: `ReportCardSchema`, `reportCardJsonSchema`, `GradeEnum`, `FindingSchema`, `CategorySchema`.

**Important:** Use `require('zod')` not `import`. Zod v4 supports CommonJS require.

<task type="auto">
  <name>Task 2: Add chatStructured function to ollama-client.js</name>
  <files>lib/ollama-client.js</files>
  <action>
Add a new `chatStructured` function to `lib/ollama-client.js` that wraps the Ollama `/api/chat` endpoint with the `format` parameter for structured output.

Function signature: `async function chatStructured(ollamaUrl, model, messages, jsonSchema, timeoutMs = 120000)`

Implementation:

1. Create an AbortController with the provided timeout.
2. POST to `${ollamaUrl}/api/chat` with body:
   ```json
   {
     "model": model,
     "messages": messages,
     "format": jsonSchema,
     "stream": false,
     "options": { "temperature": 0 }
   }
   ```
   **CRITICAL:** `stream` MUST be `false`. Structured output is unreliable with streaming (see RESEARCH.md Pitfall 1). `temperature: 0` for deterministic structured output per Ollama docs.
3. If response not ok, throw `Error(`Ollama error: ${response.status}`)`.
4. Parse response JSON, extract `data.message.content`, `JSON.parse()` it, and return the parsed object.
5. Clear timeout in finally block.

Add `chatStructured` to `module.exports`.

Follow existing code style: no semicolons at end of statements (match existing file), 2-space indent, same error handling pattern as `chatComplete`.
</action>

<task type="auto">
  <name>Task 3: Add review system prompt and fallback prompt to prompts.js</name>
  <files>lib/prompts.js</files>
  <action>
Add two new exports to `lib/prompts.js`:

**REVIEW_SYSTEM_PROMPT** — the system prompt for structured review mode. Must implement these CONTEXT.md locked decisions:

1. **Protective parent tone**: "I want to make sure this is safe for you to use" — prioritize safety warnings. Lead with what could go wrong, not praise.
2. **Analogies for serious issues only**: Use everyday analogies (e.g., "leaving your front door unlocked") for critical/high severity findings. Plain language without analogies for medium/low.
3. **Technical terms always explained**: When unavoidable, include term + parenthetical explanation: "SQL injection (when someone tricks your app into running commands on your database)".
4. **Finding length by severity**: Short (2-3 sentences) for low/medium. Longer (paragraph with analogy + action) for critical/high.
5. **Clean code celebration**: When code has no issues, give all A grades with brief positive notes per category and set cleanBillOfHealth to true.

The prompt must also describe the expected JSON structure in natural language (Ollama docs recommend this to ground the model's response alongside the format parameter).

Include grading rubric guidance (Claude's discretion area):

- A: No issues or only trivial style preferences
- B: Minor issues that won't cause problems but could be better
- C: Notable issues that should be addressed before shipping
- D: Significant problems that will likely cause issues in production
- F: Critical issues — do not ship this code

**REVIEW_FALLBACK_PROMPT** — the system prompt for chat fallback when structured output fails. Should:

- Use the same protective-parent tone
- Say something like "I couldn't generate a formal report card, but let me tell you what I found"
- Still cover bugs, security, readability, completeness in a conversational format
- Still highlight the most important thing to fix first

Add both to SYSTEM_PROMPTS object (key: 'review' for REVIEW_SYSTEM_PROMPT) and export them separately too. Add 'review-fallback' key for REVIEW_FALLBACK_PROMPT. Do NOT add MODE_GUARDRAIL to the review prompts — review mode always has code input.
</action>
