## Gaps, Risks, and Mitigations

### G1: validateKey payload validation

**Risk:** Current validateKey requires `payload.tier` and `payload.exp`. Features-based keys have `payload.features` instead of `payload.tier`.
**Mitigation:** Update validation: require `payload.exp`. Require either `payload.tier` OR `payload.features` (array, can be empty). Return both in validation result when present.

### G2: License refresh propagation

**Risk:** When user deactivates in Settings, App's licenseInfo updates via onLicenseChange→fetchLicense. But if user is currently on a builder mode panel, they can still see the UI until they navigate.
**Mitigation:** BaseBuilderPanel 403 handling (Task 6) catches the next /api/score call. For immediate UX: consider useEffect in App that when licenseInfo changes and current mode is locked, setShowUpgrade(mode) or setMode('chat').

### G3: lastMode persistence

**Risk:** lastMode saved as 'prompting'; user deactivates; next launch restores 'prompting' but it's locked.
**Mitigation:** Task 4.4 — when restoring lastMode, check isModeLocked; if locked, use 'chat'.

### G4: startTrial with feature-based model

**Risk:** startTrial currently sets tier='pro'. With features, we need features=getProFeatures().
**Mitigation:** Task 1 — startTrial, handleAppStorePurchase set features = getProFeatures(). startTrial block condition: if (tier==='pro' || (features && features.length>0)) return "Already on Pro".

### G5: generate-license-key feature validation

**Risk:** --features foo,bar could include invalid ids.
**Mitigation:** Task 2 — validate each feature against PRO_FEATURE_IDS = ['prompting','skillz','agentic','create'].

### G6: Config persistence of features

**Risk:** \_persistLicense must include features in config.license.
**Mitigation:** Task 1 — \_persistLicense adds features to updateConfig payload. On init from config, when source is appstore/trial, use config.license.features or derive getProFeatures().

## Additional Issues (Deep Analysis)

### G7: lastMode vs licenseInfo race condition — CRITICAL

**Risk:** In App.jsx init useEffect, `fetchLicense()` and `getLastMode()` run in parallel. `getLastMode().then(lastMode => _setMode(lastMode))` can run before `fetchLicense()` completes. At that moment `licenseInfo` is still `{ tier: 'free' }` (initial state). If we add lastMode fallback (Task 4.4) that checks `isModeLocked(lastMode, licenseInfo)`, the check would use stale licenseInfo.
**Mitigation:** Restore lastMode only after license is loaded. Options: (a) `await fetchLicense()` then `getLastMode()` in sequence; (b) Move lastMode restore into a second useEffect that depends on `[licenseInfo]` and runs only when licenseInfo is "ready" (e.g. after first fetch). Prefer (b): add `useEffect(() => { if (!licenseInfo) return; if (isElectron && window.electronAPI?.getLastMode) { window.electronAPI.getLastMode().then(lastMode => { if (lastMode && !isModeLocked(lastMode, licenseInfo)) _setMode(lastMode); }); } }, [licenseInfo]);` and remove lastMode from the init useEffect. This ensures lastMode is applied only when licenseInfo is available.

### G8: Empty features array semantics

**Risk:** For `features: []`, `isFeatureAllowed` should deny all pro features. Plan says "if features exists" — need to treat `[]` as "no pro features" and NOT fall back to tier.
**Mitigation:** In isFeatureAllowed: if `_licenseState.features` is defined (including `[]`), use only the features array. Do not fall back to tier when features is explicitly `[]`. Only fall back to tier when `!features || features === undefined` (legacy keys).

### G9: deactivateLicense must clear features

**Risk:** When deactivating, we set tier='free' but if we don't clear features, stale features could persist in memory. \_persistLicense replaces the whole license object, so we must set `_licenseState.features = []` in deactivateLicense.
**Mitigation:** Task 1 — in deactivateLicense, set `_licenseState.features = []` before \_persistLicense.

### G10: Trial expiry — clear features

**Risk:** When trial expires, initLicense sets tier to 'free' but doesn't enter the trial block. \_licenseState.features would be unset. Ensure we don't carry over features from a previous valid trial.
**Mitigation:** In initLicense, when trial is expired (trialDays <= 0), explicitly set `_licenseState.features = []` and do not set tier to 'pro'. The current flow already doesn't set tier when expired; add `_licenseState.features = []` for consistency.

### G11: create-mode E2E tests require license — CRITICAL

**Risk:** create-mode.spec.js does NOT mock `/api/license`. With tier='free', (1) Create button is locked — click would show upgrade modal, not wizard; (2) `request.post('/api/create-project')` returns 403 from requireTier. Tests would fail.
**Mitigation:** Add license mock to create-mode.spec.js (same pattern as builder-prompting.spec.js): `await context.route('**/api/license', route => route.fulfill({ status: 200, body: JSON.stringify({ tier: 'pro', source: 'direct', features: ['prompting','skillz','agentic','create'] }) }));` in a beforeEach or at start of each test that needs Create mode. Apply to all create-mode tests.

### G12: server.js sanitizeConfigForClient — features not stripped

**Risk:** sanitizeConfigForClient strips license to { tier, source, expiresAt }. The /api/license endpoint returns getLicenseInfo() directly and is NOT sanitized by sanitizeConfigForClient. So /api/license will include features once we add it. No change needed for /api/config — license in config is separate. Verify: GET /api/license is the source for frontend licenseInfo; it does not use sanitizeConfigForClient. Good.

### G13: SettingsPanel features display — exact behavior

**Risk:** Task 5 says "add display of features list in status". When license is feature-based (features present, possibly tier still 'free'), we need to show which features are enabled. When tier is 'pro' and no features (legacy), show "All Pro features".
**Mitigation:** In SettingsPanel License section: when `licenseData?.features?.length > 0`, display "Enabled: {licenseData.features.map(f => capitalize(f)).join(', ')}". When `licenseData?.tier === 'pro'` and no features array, display "All Pro features". Place below the "via ..." line in the status block.

### G14: isModeLocked with null/undefined licenseInfo

**Risk:** loadConversation and lastMode fallback use isModeLocked(modeId, licenseInfo). If licenseInfo is null or malformed (e.g. fetch failed), we need safe behavior.
**Mitigation:** In isModeLocked: if `!licenseInfo`, treat as `{ tier: 'free' }`. If `licenseInfo.tier` is undefined and no features, treat as free. Defensive: `const safe = licenseInfo || { tier: 'free' };`.

### G15: generate-license-key — --features with --tier

**Risk:** User might pass both --features and --tier. Need clear precedence.
**Mitigation:** Task 2 — if --features provided, use features payload (ignore --tier). Else use tier payload. Document in help text.

### G16: MCP server — no license init

**Risk:** mcp-server.js uses initConfig, initHistory but NOT initLicense. MCP tools (if any) that invoke license-gated behavior would bypass checks. MCP runs standalone, not through Express.
**Mitigation:** Out of scope for Phase 7. MCP tools currently don't go through requireTierForMode. If MCP exposes chat/code-review in future, add initLicense and feature checks. Document as future consideration.
