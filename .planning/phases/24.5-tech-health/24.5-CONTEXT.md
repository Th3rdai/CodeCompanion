# Phase 24.5: Tech Health — Decomposition, Linting & Safety

**Gathered:** 2026-04-08
**Status:** Ready for planning

<domain>
## Phase Boundary

Pure structural and quality work — no new features, no behavioral changes to existing endpoints or UI. This phase exists because two files (`server.js` at 5,169 lines, `src/App.jsx` at 2,954 lines) are approaching the size where adding Phases 25–27 (agent builtins, more routes) will start causing real maintenance pain. The objective is to bring both files below a maintainable threshold before the next feature wave, while also closing three smaller but concrete risks: direct `writeFileSync` in `history.js` that can corrupt conversations on crash, absence of a linter (style drift, undiscovered bugs), and a duplicate test tree (`tests/test/`) that confuses contributors.

**In scope:**
1. Add ESLint flat config + Prettier and format the codebase
2. Consolidate `tests/test/` duplicate tree into `tests/unit/` and `tests/e2e/`
3. Harden `lib/history.js` to use atomic write (temp file + rename), matching the pattern already in `lib/memory.js`
4. Extract App.jsx chat + model state into focused custom hooks (`useChat`, `useModels`)
5. Split `server.js` into per-domain Express sub-routers under `routes/`

**Out of scope (deferred to Phase 25+):**
- TypeScript migration
- Tool-call parser hardening / new edge-case tests (Phase 25)
- JSDoc for lib/ exports
- Any new feature or behavior change

</domain>

<decisions>
## Implementation Decisions

### Plan Sequencing

Three sequential plans, each independently verifiable:

- **Plan 01** — Linting + test consolidation + history.js atomic write: zero-risk changes, no logic touched
- **Plan 02** — App.jsx hook extraction: extract state + handlers without changing behavior; App.jsx becomes the composition layer
- **Plan 03** — server.js route decomposition: move route handlers into `routes/*.js`, server.js becomes the mount file

This order matters: Plan 01 establishes the linting baseline so Plans 02 and 03 stay clean. Plans 02 and 03 are independent of each other and could run in parallel, but Plan 01 must land first.

### ESLint / Prettier Scope

- Use ESLint flat config (`eslint.config.js`) — the modern format; avoids legacy `.eslintrc` conflicts
- Prettier for formatting only (no rule overlap with ESLint)
- Auto-fix on first run; commit formatted code as a single "chore: format codebase" commit so future diffs stay readable
- No TypeScript rules — `@typescript-eslint` is out of scope; JS rules only
- Rule set: `eslint:recommended` + `react` plugin for JSX; no aggressive stylistic rules beyond Prettier

### Test Consolidation

- `tests/test/unit/icm-scaffolder.test.js` → move to `tests/unit/`
- `tests/test/e2e/create-mode.spec.js` → move to `tests/e2e/`
- Delete `tests/test/` tree
- Update any script references if needed; Playwright config already discovers `tests/` so both were being discovered — post-consolidation only the canonical paths remain

### History Atomic Write

- `lib/history.js` line 121: `fs.writeFileSync(path.join(_historyDir, id+'.json'), jsonString)`
- Replace with: write to `id.json.tmp`, then `fs.renameSync` to `id.json` (exact pattern from `lib/memory.js` lines 207–208)
- `lib/memory.js` is already correct — no change needed there

### App.jsx Hook Extraction — What Moves Where

**`useChat` hook** (`src/hooks/useChat.js`):
- State: `messages`, `streaming`, `activeConvId`, `history`, `showArchived`, `renaming`, `sendBurst`, `pendingAutoSend` ref, `chatAbortRef`, `chatTokenRafRef`
- Handlers: `handleSend`, `handleStop`, `handleNewChat`, `handleLoadConversation`, `handleDeleteConversation`, `handleBatchDelete`, `handleSaveChat`
- Returns everything App.jsx needs to pass down to chat UI and sidebar

**`useModels` hook** (`src/hooks/useModels.js`):
- State: `models`, `connected`, `selectedModel`, `ollamaUrl`, `autoResolvedLabel`

- Returns model list + selection handlers

App.jsx keeps: mode state, panel visibility flags, settings, toast, file attachments, router-style `currentPanel()` render logic.
Target: App.jsx under 2,000 lines after extraction.

### Shared Dependencies Between Routers

Several lib modules are used across many routers. Rather than re-requiring in each file, server.js will pass a lightweight `appContext` object at mount time:
- `appContext.config` — config module
- `appContext.ollamaUrl()` — getter for current URL
- `appContext.limiter.*` — rate limiters (chat, create, score, etc.)
- `appContext.requireLocalOrApiKey` — middleware function

Routers receive `appContext` via a factory function: `module.exports = function createRouter(appContext) { const router = express.Router(); ... return router; }`. **Important**: rate limiters are registered as `app.use('/api/route', createRateLimiter(...))` in server.js and remain there — they are app-level middleware that run before any router, share in-memory buckets, and must NOT be moved into individual routers. `appContext` therefore contains only `{ config, requireLocalOrApiKey }`.

<specifics>
## Specific Ideas

- The `routes/chat.js` file will be the most complex extraction (~810 lines) because `/api/chat` holds the full agent tool loop and SSE streaming. It must carry its own imports for `tool-call-handler`, `builtin-agent-tools`, `memory`, `history`, `ollama-client`. The chat abort controller stays inside the route handler closure — it does not need to live in server.js.
- Keep rate limiter definitions in server.js and pass them via `appContext.limiter` so the same in-memory buckets are reused across all routers (a fresh limiter per-router would defeat the point).
- After Plan 03, run the full test suite to verify no import paths broke. The integration tests spin up real server instances, so they will catch missing route registrations.
- Do NOT move the `app.get('*')` SPA fallback — it must remain last in server.js, after all sub-router mounts, to preserve current route priority.

</specifics>
